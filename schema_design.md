# 1. Schema Structure and Design Rationale
* The metadata schema is organized into five sequential layers, each addressing different types of information that I found to be absent from the HDF5 files during Task 1 analysis. I have tried to order the layers based on how I thought a physicist would document their analysis, moving from the broadest context (what experiment, what process) down to the most specific technical details (how the neural network was trained).
* I have kept the schema as a single YAML file for the entire dataset instead of one YAML file per HDF5 file. I took this decision because in Task 1, it was found that the three files are not independent. They are related systematic variations of the same measurement. A per-file schema would have repeated the dataset identity, simulation, and event selection information three times without showing the relationship between the files. I have instead showed the relationship explicitly through the files section, where each file is assigned a role (nominal or systematic_variation) and a variation_type, making the hierarchy/relationship immediately clear to any user.
* A lot of the fields whose information was not provided in the HDF5 files has been kept as "NULL" in the YAML file and can be filled in by the physicist originally performing the experiment as they would have those data available to them.

## The five layers and their rationale

* Layer 1 (Dataset Identity)- captures the minimum information needed to identify what the dataset is, such as- experiment, physics process, collision energy, and publication identifiers. The doi and hepdata_id fields are included as null placeholders and can be filled in at the time of publication.
* Layer 2 (Simulation)- documents how the Monte Carlo simulation was generated. The generator, parton shower program, PDF set, and tune are all critical for theory comparisons and two analyses using the same generator but different tunes will produce different predictions. This layer only documents the nominal simulation, and as I have stated previously, the systematic generator variations are handled in Layer 4 under each file's entry.
* Layer 3 (Event Selection)- records the kinematic cuts applied before unfolding. Without knowing what selection criteria were applied, a physicist cannot determine whether their events are compatible with the published weights.
* Layer 4 (File Inventory)- forms the core of the schema. It documents every column in every file with its name, description, physical units, and usage instructions. It also provides how to use each weight type. For example, the ensemble weights section explains that uncertainty bands are computed by applying each of the 100 weights and taking the standard deviation across results.
* Layer 5 (OmniFold Provenance)- stores the training configuration and software environment. This layer is split into required fields (number of iterations, ensemble size, input features) and optional subsections (model architecture, software versions). Without the optional metadata, we can still reuse the dataset by applying the weights to compute a new observable, but with the optional metadata, we can have full reproducibility by re-running the entire training from scratch. While a user who only wants to apply the published weights might not need the neural network architecture, a user who wants to verify or extend the training does.

# 2. What Was Included vs. Left Out
## Included
* Usage instructions embedded in weight descriptions. Rather than simply naming each weight column, the schema explains how each weight type should be applied. This was included due to the observation that the 176 weight columns in the nominal file span five functionally distinct subcategories: base weights, ensemble members, bootstrap replicas, data-driven weights, and systematic variations. All of these serve entirely different purposes and must be used differently. A user encountering this file without prior OmniFold knowledge would have no way to distinguish between them from column names alone.
* A type field on every systematic weight. Each of the 22 systematic weight columns is tagged with a type: efficiency, calibration, theory, normalization, or background. This allows downstream tools to automatically group systematics for envelope calculations without requiring the user to manually categorize them. It also makes the schema more useful for automated HEPData ingestion pipelines.
* Explicit documentation of target_dd ambiguity. The target_dd column was found during Task 1 analysis to contain continuous float values despite its name implying a binary classifier label. Rather than guessing its meaning or omitting it, the schema documents the observed value range and explicitly flags that the exact interpretation must be provided by the analyst.
* Cross-file structural differences explicitly documented. Task 1 revealed that the three files have substantially different column counts (200, 51, and 26 respectively). The schema captures this by listing each file's weight columns individually rather than assuming a common structure. A user can immediately see that multifold_nonDY.h5 contains only the bare minimum weights while multifold.h5 contains the full uncertainty decomposition.

## Left out
* Bibliographic metadata (authors, paper title, journal). Author attribution and publication details are the domain of HEPData. Including them in a technical metadata schema would create redundancy and a maintenance burden. If an author list changes after submission, the schema would need to be updated separately from HEPData. The hepdata_id field provides a pointer to the full bibliographic record without duplicating it.
* Detailed statistical summaries of observable distributions. While Task 1 computed means and standard deviations for several observables, these were not included in the schema. Observable distributions are analysis-specific and can be computed by any user from the HDF5 files directly.
* Explicit data/MC row labeling as a separate column specification. The schema documents the data_mc_split field in the OmniFold provenance section rather than prescribing a specific column format. Different analyses handle data/MC mixing differently (some use separate files, some use a label column, some use an index range), and the schema should accommodate all approaches rather than mandating one.

# 3. How a New User Would Interact With This File
A physicist who did not run the original analysis and wants to reuse the published OmniFold weights would interact with the metadata schema in a sequential, top-down manner. The following describes the expected workflow:
* Orientation: The user first reads the dataset section to understand what measurement this is, what experiment produced it, and where the full publication can be found via the doi and hepdata_id fields. This immediately tells you what you are looking at.
* Compatibility check: The user reads the event_selection section to verify that their own events or theory predictions are compatible with the published weights. If they are trying to apply the weights to a new Monte Carlo sample, they need to confirm that their sample passes the same kinematic cuts.
* Loading the right file: The user reads the files section to identify which of the three HDF5 files they need. If they want the primary result, they load multifold.h5 and apply weights_nominal multiplied by weight_mc. If they want to assess generator uncertainty, they compare weights_nominal between multifold.h5 and multifold_sherpa.h5.
* Computing a new observable: The user reads the ensemble section to understand how to compute statistical uncertainty bands on their observable of choice. The schema tells them to apply each of the 100 ensemble weights, compute their histogram, and take the standard deviation.
* Assessing systematic uncertainties: The user reads the systematics section to identify which systematic weights are relevant to their analysis. The type field lets them quickly filter to, for example, only theory systematics or only efficiency systematics, without having to decode column names manually.
* Reproducibility: A user who wants to re-run the unfolding reads the omnifold section to find the training configuration.
